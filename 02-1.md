# 프롬프트 엔지니어링

## 1. Zero-shot

정의 : 모델에게 예시를 주지 않고 작업 설명만 전달해 결과를 요청하는 프롬프트, 출력 형식이 흔들린다.

비유 : 신입사원 "보고서 써 주세요", 처음 업무 맡기는 상황, 회사가 원하는 양식, 분량, 말투, 결론의 위치 모르면 결과가 사람마다 달라진다.

## 2. Few-shot

정의 : 모델에게 새 과제를 바로 시키기 전에 입력과 원하는 출력의 예시를 2~3개 정도 먼저 보여주는 프롬프트

비유 : 신입사원 "고객 리뷰를 분류 해 주세요." -> 3점 척도, 5점 척도, 긍정/부정 -> 이 리뷰는 긍정, 저 리뷰는 부정으로 분류 해주세요.

- 부정 : 영화가 재미 없어요.
- 부정 : 영화가 지루해요.
- 긍정 : 시간 가는 줄 몰랐어요.

## 3. Chain - of - Thought

정의 : 모델에게 최종 답변만 바로 내라고 하지 않고, 문제를 단계적으로 생각하도록 유도하는 프롬프트

비유 : 수학시험에서 정답만 쓰시오, -> 학생이 어디서 틀렸는지 확인하기 어렵다 -> 풀이 과정을 쓰시오 -> 계산 순서가 드러나서 -> 계산 순서가 드러나고 오류 지점을 찾을 수 있다.

---

프롬프트는 AI 에게 주는 업무 지시서 입니다. '보고서 써' 라고 하면 결과가 천차만별로 나온다.

'지난번처럼 이런 형식으로 써'라고 -> Few-shot 이 된다.

'먼저 자료 분석하고, 핵심 정리하고, 그 다음 보고서를 써' -> CoT

## 4. ReAct

정의 : 답변을 바로 내기 전에 Thought, Action, Observation, 같은 단계 라벨을 통해 문제 분석과 확인 과정을 분리해서 보는 방법.

- Thought: 이 단서가 무엇을 의미 하는지?
- Action: 무엇을 확인해야 하는지?
- Observation: 확인해 보니 이런 결과가 나왔네.

# 7. PCT (Persona, Context, Task)

정의 : 시스템 프롬프트 (모델에게 줄 업무 지시서) -> Persona, Context, Task 세 부분으로 나누어 작성하는 방법

- Persona : 모델이 어떤 전문가처럼 말해야하는지 지정.
- Context: 지금 사용자가 어떤 상황에 있는지 알려주는거.
- Task: 어떤 결과를 기준으로 내야하는지 제한.

# 8. Silent Instruction Conflicts

오류 없는 충돌 : 서로 충돌하는 지시가 있을 때 오류가 나지는 않는다, 모델이 일부 지시를 조용히 무시하는 현상?, 답변의 기준을 흔들리게 만드는 상황.

- Task : 한 문장으로만 답변하세요.
- Persona : 근거를 상세하게 설명하세요.

# 9. 프롬프트 인젝션

- sql 인젝션, cors(Cross Origin Resource Sharing)
- 정의 : 사용자가 시스템 지시를 공개하거나 무력화 하려는 입력을 넣었을 때 모델이 기존 지시 우선 순위를 유지하고 위험 요청을 거절 하도록 설계하는 프롬프트 안전 장치.

# 10. Prompt Caching

정의 : 동일하거나 반복되는 큰 시스템 프롬프트를 여러번 사용할 때 입력 토큰 비용이 크게 낮아질 수 있는 방법. 비용 최적화 관점.
